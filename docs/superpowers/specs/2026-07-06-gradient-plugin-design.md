# gradient — Claude Code Plugin Distribution — Design

**Date:** 2026-07-06
**Status:** Draft (brainstorming complete; awaiting user review)
**Scope:** Spec 5. Packages gradient itself as a Claude Code plugin so
`/plugin install gradient` replaces `npx gradient` + `gradient init` as the
recommended onboarding. Distinct from Spec 4 Phase E (`gradient bundle`),
which packages the *user's mined artifacts*; this spec packages the *tool*.

---

## 1. Context

Adoption of gradient itself is the outermost funnel leak — upstream of every
leak Spec 4 closes. Today onboarding is `npx gradient scan` plus an optional
user-level skill (`cli/src/skill/SKILL.md`, installed by `init --skill`) that
assumes `gradient` is on PATH. Claude Code's plugin system is where users
already discover tooling: a plugin ships skills, executables (`bin/`), and a
manifest, installs with one command, and updates through git versioning. A
GitHub repo can double as a marketplace via `.claude-plugin/marketplace.json`.

Ecosystem check (July 2026): plugins can also ship hooks, MCP servers, LSP
servers, monitors, and default settings. gradient deliberately uses none of
those surfaces (§2 #1, §8) — a trust-first tool must be inert on install.

## 2. Decisions (locked during brainstorming)

| # | Decision | Choice |
|---|----------|--------|
| 1 | Hooks | **No hooks in the plugin manifest.** Autopilot / recall / session-scan hooks keep being installed per project into `settings.json` by their opt-in commands, exactly as today. `/plugin install` is completely inert until the user turns something on — "nothing runs without you turning it on" stays literally true. |
| 2 | Contents | **Skills + bundled CLI + manifest + README only.** No agents, MCP servers, LSP, monitors, or default settings. Every one of those surfaces either runs code or changes behavior on install. |
| 3 | Binary | **`plugin/bin/gradient.js` — a committed single-file ESM bundle** built by esbuild (devDependency; the one runtime dep, `@anthropic-ai/sdk`, is inlined). Marketplace installs pull from git, so the bundle must be committed; `npm run build:plugin` regenerates it and is a release step. Node ≥ 20, invoked as `node "${CLAUDE_PLUGIN_ROOT}/bin/gradient.js" …` — no PATH or exec-bit reliance. |
| 4 | Skills | **Four launch skills**: `gradient:scan`, `gradient:review`, `gradient:stats` (model-invocable), and `gradient:autopilot` (`disable-model-invocation: true` — authority changes are typed by the user, never chosen by the model). |
| 5 | CLI surface | **One CLI addition: `gradient review --json`** — a non-interactive listing of cached suggestions for the review skill to present in-chat. The readline flow is unchanged and remains the terminal path. |
| 6 | Versioning | **`cli/package.json` is the single version source.** `build:plugin` writes `plugin.json`'s `version` from it; a test asserts they match. |
| 7 | Distribution | **Repo-as-marketplace first.** Root `.claude-plugin/marketplace.json` makes `/plugin marketplace add ylambda/gradient` work immediately. Community/official marketplace submission is a manual release step, out of scope for the CLI. |
| 8 | npx path | **Retained.** `npx gradient` and `init --skill` stay supported; docs present the plugin as the recommended install. The two may coexist (a user-level `gradient` skill plus namespaced `gradient:*` skills) — noisy but harmless; the plugin README tells plugin users to skip `init --skill`. |

## 3. Layout

```
gradient/
  .claude-plugin/
    marketplace.json            # repo doubles as a marketplace; points at plugin/
  plugin/
    .claude-plugin/plugin.json  # {name:"gradient", description, version} — version synced from cli/package.json
    bin/gradient.js             # committed esbuild bundle (regenerated by build:plugin)
    skills/
      scan/SKILL.md
      review/SKILL.md
      stats/SKILL.md
      autopilot/SKILL.md
    README.md                   # what installs, what does NOT run, provenance
  cli/                          # unchanged home of the source
```

`plugin.json` carries only `name`, `description`, `version` (plus `author`,
`homepage`) — the minimum verified against current plugin loading (open
question, shared with Spec 4 Phase E).

## 4. Skills

Every skill body invokes the bundled binary explicitly:

```
node "${CLAUDE_PLUGIN_ROOT}/bin/gradient.js" <subcommand> …
```

- **`gradient:scan`** — runs `scan` (project scope by default; `--user` /
  `--all` only when the user asks), then summarizes the printed suggestions.
  Notes that scan is read-only.
- **`gradient:review`** — runs `review --json`, presents the suggestions
  (name, title, type, evidence, confidence), lets the user choose, then runs
  `apply <id>` for each approved one and reports written paths. **Never
  applies without an explicit user choice in the conversation** — same stance
  as the existing `init --skill` skill.
- **`gradient:stats`** — runs `stats` (and `insights` once Spec 4 Phase D
  ships) and relays the output.
- **`gradient:autopilot`** — `disable-model-invocation: true`. Explains the
  modes, then runs exactly the subcommand the user asked for
  (`status` / `nudge` / `full` / `off`).

Failure mode: if the bundle is missing or fails to execute, each skill
instructs Claude to tell the user to reinstall the plugin — skills never fall
back to a PATH `gradient` (that would blur which install is running).

### `review --json` output

An array mirroring `Suggestion` (id, name, title, rationale, evidence,
confidence, payload), read from the same `loadSuggestions` used by `review`.
No new information is exposed — it is the cached `suggestions.json` with
stable field names for conversational rendering.

## 5. Build

- `npm run build:plugin` (in `cli/`): esbuild bundle of `src/cli.ts` →
  `../plugin/bin/gradient.js` (`platform: node`, `format: esm`,
  `target: node20`, minify off — the bundle is reviewable in diffs), then
  rewrite `plugin/.claude-plugin/plugin.json` with the package version.
- esbuild is a pinned devDependency; the runtime-dependency count stays at
  one (`@anthropic-ai/sdk`, inlined into the bundle).
- Release checklist gains: bump version → `build:plugin` → commit bundle +
  manifest together. A stale bundle is detectable: the version-sync test
  fails when `plugin.json` ≠ `package.json`.

## 6. Recursion / in-session behavior

Running `scan` from inside a Claude Code session (via the Bash tool) spawns
`claude -p` for the detect step. That is a plain subprocess, not a hook
cycle — the autopilot recursion guard (Spec 2) is `respond`-specific and
stays untouched. The scan skill keeps default project scope so in-session
runs stay fast; `--detach` remains available when the user wants it out of
the way.

## 7. Testing

- **Inert-on-install guard:** a test asserts `plugin/` contains no `hooks/`
  directory, no `.mcp.json`, no `settings.json` — the §2 #1 promise, encoded.
- Version sync: `plugin.json` version === `cli/package.json` version.
- Bundle smoke: spawn `node plugin/bin/gradient.js --version` (no network).
- `review --json`: golden output against a fixture `suggestions.json`; empty
  cache → `[]`, exit 0.
- Skill frontmatter: every SKILL.md has `description`; autopilot has
  `disable-model-invocation: true`; scan/review/stats do not.
- `marketplace.json`: parses, points at `plugin/`.

## 8. Explicitly out of scope (YAGNI)

- **Hooks, MCP, LSP, monitors, default settings in the plugin** — Decision 1/2.
- **Marketplace submission automation** — manual release step.
- **Auto-update logic** — git-based plugin versioning already handles it.
- **Deprecating the npx flow or `init --skill`** — retained (Decision 8).
- **Bundling user artifacts** — that is Spec 4 Phase E, unchanged.

## 9. Sequencing & dependencies

Independent of Spec 4's phases: the skills wrap already-shipped commands.
`review --json` touches `commands/review.ts` only additively. When Phase D
(`insights`) ships, the stats skill body gains one line — recorded as a
follow-up task in the Phase D plan, not a dependency here.

## 10. Dead code & outdated content (handled as part of execution)

- `cli/src/skill/SKILL.md` is **retained deliberately** (Decision 8 — the
  npx path's skill), but its body must not drift from the plugin skills'
  command wording; the plan adds a wording pass.
- Root `README.md` Quickstart gains the plugin install as the first path;
  `cli/README.md` stays npx-focused. No removals.
- `plugin/README.md` states explicitly: installing runs nothing, writes
  nothing outside the plugin dir, and every automation remains opt-in via
  CLI commands.

## 11. Open questions for the implementation plan

- Verify `${CLAUDE_PLUGIN_ROOT}` expansion inside SKILL.md bodies against
  current Claude Code docs at build time (fallback: plugin `bin/` PATH
  injection, which would require committed exec bits).
- Minimum required `plugin.json` fields for current plugin loading
  (`name`, `description`, `version` assumed) — same open question as Spec 4
  Phase E; whichever plan lands first pins it.
- Whether `marketplace.json` requires an `owner` field and its exact schema.
- esbuild version pin and whether the committed bundle needs a lint-ignore
  entry.
